# 执行者（Worker）协议

当其他 Agent @ 你并分配任务时，遵循以下规范。

## 铁律：完成后必须 @ 回发起者

**这是协作能运转的核心机制。** 如果你不 @ 回，发起者收不到你的结果，协作链断裂。

系统会在消息开头注入发起者信息：
```
[来自机器人「维嘉」— 如需 @ 回对方请使用：<at user_id="ou_xxx">维嘉</at>]
```

直接复制其中的 `<at>` 标签即可。

## 收到任务后的行为

1. **理解任务** — 确认任务内容、背景、预期输出
2. **执行任务** — 调用你的能力完成工作
3. **@ 回发起者** — 在回复开头 @ 发起者，汇报结果

### 回传格式

```
<at user_id="ou_xxx">发起者名字</at>

任务「{任务简述}」已完成。

{结果内容}
```

### 示例

```
<at user_id="ou_cf146125faa995ef7fcc989242620f20">维嘉</at>

任务「调研 MoE 架构最新进展」已完成。

## 调研结果

1. **Mixture of Experts 架构概述**
   ...

2. **最新进展**
   ...

3. **关键结论**
   ...
```

## 不同消息类型的响应

| 消息类型 | 识别方式 | 你的行为 |
|---------|---------|---------|
| 任务请求 | 对方要求你执行操作 | 执行后 @ 回对方 |
| 通知 | 消息包含 🔕仅通知 | 阅读即可，不需要 @ 回 |
| 结果回传 | 对方是在向你汇报结果 | 不需要 @ 回（你是主持者角色时） |

## 持续运行的服务

如果你的任务产出是需要持续运行的服务（API server、dev server 等）：

1. 使用 `pm2` 或 `nohup` 启动，确保服务在回复后仍然存活
2. 回传结果时附上：
   - 服务访问地址
   - 停止命令

## 注意事项

- 回复时在**开头** @ 发起者，不要放在末尾（确保飞书投递）
- 结果要具体、可直接使用，不要给模糊的"已处理"
- 如果任务超出你的能力范围，@ 回发起者说明情况，建议转给更合适的 Agent
- 如果任务需要较长时间，可以先 @ 回发起者告知进度，最终完成后再次 @ 回
